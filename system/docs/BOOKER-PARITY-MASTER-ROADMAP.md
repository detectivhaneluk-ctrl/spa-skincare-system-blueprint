## 3. Locked rules / forbidden moves (“Do Not Do”)

1. **Do not do broad refactors** (framework change, ORM, renaming half the codebase, “cleanup” unrelated to the task).
2. **Do not rewrite architecture without proof** — any structural change needs inventory, contract update, and explicit approval in a **single** focused task.
3. **Do not mix unrelated modules in one task** (e.g. payroll + public booking in one PR).
4. **Do not skip runtime / root-cause verification** — reproduce with logs, SQL, or scripted smoke before claiming fix (see `BACKEND-READINESS-QA.md`).
5. **Do not jump to mobile, AI, marketplace, or speculative “all channels” work** before the **§5.C** priorities are consciously addressed or deferred with written rationale.

### 3.1 Status vocabulary (mandatory)

All roadmap and OPS status claims must use **only** these labels (including cross-links to `TASK-STATE-MATRIX.md`):

- `CLOSED` = runtime-verified and behaviorally closed (historical rows may say “shipped” with the same meaning).
- `PARTIAL` = implementation exists; invariant, end-to-end bar, or full operational depth is **not** closed.
- `OPEN` = not implemented, not started, or no production-grade closure (includes “proof gap” cases—treat as `OPEN` until a fail-closed automated proof exists).
- `REOPENED` = previously treated closed; evidence shows material remaining risk (e.g. universal tenant fail-closed, full lifecycle/suspension enforcement).
- `AUDIT-ONLY` = read-only verifier, truth map, or audit memo **only**—**not** a write-path or runtime product closure unless separately `CLOSED`.
- `PLANNED` = committed direction / charter text without repo proof yet (often overlaps `OPEN` for execution).

No `AUDIT-ONLY` or read-model-only item may be labeled `CLOSED` as if the full domain were finished.

## 5. Active plan (canonical — rebased full-ZIP audit)

**Strict product / Booker-parity execution queue = §5.C** (macro-phases **1–5** with ordered task IDs: stabilization → operational depth → native catalog → mixed sales **implementation** → storefront). Subsections **D–F** are hold, archive, and doc-truth hygiene. When other docs say “§5”, they mean **this section**; **§5.C** is the prioritized **feature** backlog.

**Platform / SaaS / ops spine = §6** (maintainability, tenancy, packages, onboarding/offboarding, scale). It does **not** override §5.C sequencing unless a §6 task explicitly gates risk (document that in the task).

### A. Current system stage

The repository is a branch-scoped salon operations backend with mature appointments (including public booking APIs, waitlist automation, and recurring series), sales/invoicing/payments, memberships lifecycle and renewal billing, documents and consents, intake forms, an outbound notification queue with dispatch script, staff-facing marketing campaigns, payroll commission runs, and **anonymous public commerce** (catalog, initiate, finalize → **`awaiting_verification`** without anonymous `PaymentService::create` from client payment fields — see ```463:581:system/modules/public-commerce/services/PublicCommerceService.php```). **Services** today use **flat** `service_categories` (migration `019`: name/sort/branch only — **no parent/child**); `services.category_id` FK is single-level. **Products** (`products`, migration `030`) use **string** `category` and `brand` columns — not first-class hierarchical categories or brand entities. **Planned expansion** (§5.C Phases **3–5**): hierarchical service categories, richer service catalog fields, brand + cosmetic/product category foundation, **unified native catalog domain** (this app as single source of truth), **mixed service+retail invoices**, then **storefront exposure** — without WooCommerce sync as authority.

### B. Proven completed foundations (short list, code-backed)

- **Core ops:** Appointments, availability, staff schedules, clients/CRM, sales/invoices/payments/register, inventory movements, JSON reports (`system/modules/*`, `system/routes/web.php`).
